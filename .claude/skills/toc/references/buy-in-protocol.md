# Layers of Resistance (Buy-in) Protocol

Systematically overcomes resistance to change by addressing each layer
of objection in the correct sequence.

From Goldratt's "It's Not Luck" (1994) and further developed in TOC practice.

## When This Tool Applies

- A proposed change faces pushback from stakeholders
- People agree with the analysis but won't act
- "Yes, but..." responses to every suggestion
- Need to build consensus for a difficult decision
- Presenting TOC analysis results to skeptics

## Core Insight

People resist change not because they're irrational, but because they have
legitimate concerns that haven't been addressed. These concerns follow a
predictable sequence — skip a layer, and everything built on top collapses.

---

## The Six Layers

Each layer must be resolved BEFORE the next can be addressed.
Like building a house: foundation first, roof last.

```
Layer 0: We agree on the PROBLEM               (CRT provides this)
Layer 1: We agree on the DIRECTION              (EC provides this)
Layer 2: We agree the SOLUTION will work        (FRT provides this)
Layer 3: We can handle the NEGATIVE effects     (FRT NBR trimming)
Layer 4: We can overcome the OBSTACLES          (PRT provides this)
Layer 5: We have UNVERBALIZED FEARS addressed   (needs direct conversation)
```

---

## Layer 0: Agreement on the Problem

### The Resistance

"I don't think there IS a problem."
"That's not our real problem."
"The problem is something else entirely."

### Diagnosis

Active when:
- Stakeholder dismisses or minimizes symptoms
- "We've always done it this way and it's fine"
- Different people identify different problems
- No shared understanding of UDEs

### Strategy

Use the CRT to demonstrate:
1. List UDEs that the stakeholder PERSONALLY experiences
2. Show the cause-effect chain connecting THEIR concerns
3. Reveal the root cause that explains ALL the symptoms

### Talking Points

```
"Would you agree that [UDE] is happening?"
"And would you agree that [UDE] is also a problem?"
"Here's what connects them..."
"The common root cause is [X]. Does that resonate with your experience?"
```

### Success Criterion

Stakeholder says: "Yes, THAT is the real problem."

---

## Layer 1: Agreement on the Direction of the Solution

### The Resistance

"Okay, it's a problem, but your solution is wrong."
"That won't work in our situation."
"I have a better idea."

### Diagnosis

Active when:
- Problem is acknowledged but proposed direction is rejected
- Alternative solutions are suggested that address symptoms, not root cause
- "We need to [symptom fix]" instead of addressing the core conflict

### Strategy

Use the EC to demonstrate:
1. Show the underlying conflict (the REAL dilemma)
2. Surface the hidden assumptions
3. Show which assumption breaks
4. Present the injection as a logical consequence of breaking the assumption

### Talking Points

```
"The real dilemma is: we think we must choose between [D] and [D']."
"But that assumes [broken assumption]."
"What if that assumption isn't always true?"
"If we [injection], we don't have to choose — we get both."
```

### Success Criterion

Stakeholder says: "That's an interesting approach. It COULD work..."

---

## Layer 2: Agreement that the Solution Will Work

### The Resistance

"I see the logic, but will it really produce the results you claim?"
"In theory maybe, but in practice..."
"Show me the evidence."

### Diagnosis

Active when:
- Direction is tentatively accepted but skepticism about outcomes
- Requests for proof, case studies, or pilots
- "I need to see it work first"

### Strategy

Use the FRT to demonstrate:
1. Trace the injection forward to specific Desirable Effects
2. Map each DE back to an original UDE
3. Show the logical chain: If injection → then DE1, DE2, DE3...
4. Provide evidence from similar implementations if available

### Talking Points

```
"If we [injection], the direct effect is [DE1]."
"That leads to [DE2], which means [DE3]."
"Specifically, [UDE from our CRT] turns into [corresponding DE]."
"Here's where this has worked before: [evidence/case]."
```

### Success Criterion

Stakeholder says: "Okay, I can see how it would work, BUT..."

---

## Layer 3: "Yes, but..." — Handling Negative Side-Effects

### The Resistance

"Yes, but what about [negative consequence]?"
"You haven't thought about [risk]."
"This will cause [new problem]."

### Diagnosis

Active when:
- Solution is accepted in principle but specific risks are raised
- Every "yes, but..." is a Negative Branch Reservation (NBR)
- Concerns may be legitimate OR may be disguised resistance

### Strategy

Use FRT NBR analysis:
1. Acknowledge each "yes, but..." as a legitimate concern
2. Show you've already identified it (NBR in the FRT)
3. Present the trim that prevents the negative branch
4. If it's a new concern, add it to the FRT and trim it

### Talking Points

```
"You're right — [NBR] is a real risk."
"We've thought about that. Here's how we prevent it: [trim]."
"With [trim] in place, [NBR] doesn't materialize because [reason]."
"Does that address your concern?"
```

### Key Principle

NEVER dismiss a "yes, but..." — ALWAYS treat it as legitimate.
Even if it feels like resistance, acknowledging it builds trust.

### Success Criterion

Stakeholder runs out of "yes, but..." objections.

---

## Layer 4: "We can't because..." — Overcoming Obstacles

### The Resistance

"We can't do this because [obstacle]."
"We don't have [resource/skill/authority]."
"[External factor] prevents us."

### Diagnosis

Active when:
- Solution is accepted but implementation seems impossible
- Practical barriers are cited
- "Great idea, but we can't execute it"

### Strategy

Use the PRT to demonstrate:
1. List each obstacle they raise
2. Show the intermediate objective that overcomes it
3. Present the implementation sequence
4. Highlight quick wins that can start immediately

### Talking Points

```
"You're right, [obstacle] is real."
"To overcome it, we need to [intermediate objective]."
"Here's the sequence: first [IO1], then [IO2], then [IO3]."
"We can start with [quick win] tomorrow — no approval needed."
```

### Success Criterion

Stakeholder says: "I guess we could try..."

---

## Layer 5: Unverbalized Fears

### The Resistance

Silence. Vague discomfort. "Let me think about it."
"I'm not sure about the timing."
"We should wait until [indefinite future event]."

### Diagnosis

Active when:
- All logical objections have been addressed
- Person STILL doesn't commit
- Evasive responses, delayed decisions
- Body language says no, words say maybe

### Common Hidden Fears

| Fear | Signal |
|------|--------|
| Loss of power/status | "My role would change" |
| Loss of expertise value | "I've spent years mastering this process" |
| Career risk | "What if it fails and I championed it?" |
| Comfort zone | "I know the current system works" |
| Trust deficit | "I've seen initiatives fail before" |
| Personal impact | "This affects my team/reports/workload" |

### Strategy

This layer CANNOT be addressed with logic. It requires:

1. **Empathy**: Acknowledge the emotional reality
2. **Safety**: Reduce the personal risk of supporting the change
3. **Inclusion**: Make them part of the solution, not a target of it
4. **Small wins**: Let them experience success before full commitment
5. **Peer proof**: Others who feared and succeeded

### Talking Points

```
"I can see you have concerns beyond the logistics. What worries you most?"
"Your expertise is actually MORE valuable in the new approach because [reason]."
"What if we piloted this in a small area first? You'd lead it."
"[Peer] had similar concerns and found that [positive outcome]."
"What would make you comfortable moving forward?"
```

### Success Criterion

Stakeholder says: "Okay, let's do it."

## Integration with Other TOC Tools

| Layer | TOC Tool | What It Provides |
|-------|----------|-----------------|
| 0 | CRT | Proof of the problem (UDEs + root cause) |
| 1 | EC | Direction of the solution (broken assumption + injection) |
| 2 | FRT | Proof the solution works (DEs mapped to UDEs) |
| 3 | FRT | NBR trims (handled side-effects) |
| 4 | PRT | Implementation plan (obstacles + IOs) |
| 5 | — | Requires human empathy, not logic tools |

The full TOC analysis (`/toc`) produces the ammunition for Layers 0-4.
Layer 5 requires the human touch.

---

## Anti-Patterns

| Anti-Pattern | Why It's Wrong | Fix |
|-------------|---------------|-----|
| Jumping to Layer 2 | Skipping problem agreement = building on sand | Start from Layer 0, ALWAYS |
| Arguing against resistance | Resistance is information, not opposition | Acknowledge, then address |
| Dismissing "yes, but..." | Each concern is a legitimate NBR | Treat every objection as real |
| Presenting data at Layer 5 | Emotional concerns need emotional responses | Empathy, not spreadsheets |
| Assuming silence = agreement | Silence at Layer 5 = unresolved fear | Gently probe: "What concerns remain?" |
| One presentation for all layers | Each layer needs different content | Build the argument layer by layer |
